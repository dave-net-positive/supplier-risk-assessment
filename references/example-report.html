Questions for Example Supplier Ltd

These questions are written so that a non-technical contact can forward them without interpretation, while giving the supplier's technical team all the detail they need to respond.

Q1 Unpatched vulnerability on public-facing server Critical Confirm and fix

Our security monitoring has identified a known, publicly documented vulnerability on one of your internet-facing servers. A fix has been available since 2023 and there is publicly available code that could be used to exploit this flaw. Could you confirm whether you are aware of this, whether it affects the infrastructure that would serve the organisation, and when you expect it to be resolved?

Technical reference: CVE-2023-12345 — Apache HTTP Server 2.4.50 — observed on app.examplesupplier.com and portal.examplesupplier.com
Observed on: all visible customer subdomains (platform default). This appears to reflect the supplier's standard server configuration.
Q2 Secure connection and browser protection settings High Confirm and fix

Two standard browser security settings are not enabled on your platform. The first tells browsers to always use an encrypted connection. The second gives browsers rules about which content to trust, preventing malicious code from running on your pages. Both are straightforward to enable. Could you confirm these will be addressed and provide a target date?

Technical reference: (1) HTTP Strict-Transport-Security (HSTS) — absent all subdomains. (2) Content-Security-Policy (CSP) — absent app.* and portal.*
Observed on: all or nearly all visible customer subdomains (platform default).
Q3 Older encryption and session security on one subdomain High Clarify scope

On one subdomain we observed an older, less secure encrypted connection method, retired as unsafe several years ago. Session tokens on this subdomain are also not marked as secure. This appeared on only one of approximately 40 subdomains, suggesting a legacy deployment. Could you confirm whether this subdomain is still active, whether it would be part of any the organisation deployment, and whether these issues apply to your standard platform?

Technical reference: (1) TLS 1.0 / RC4 cipher — legacy.examplesupplier.com. (2) SessionId cookie Secure flag absent — legacy.examplesupplier.com.
Observed on: 1 of ~40 visible subdomains (outlier). Likely a legacy deployment; included for confirmation.
Q4 Email domain configuration Low Confirm and fix

Your domain's email settings allow any server to send email claiming to be from your organisation, making phishing emails easier to construct. This is a simple DNS configuration change. Could you confirm this will be tightened and provide a timeline?

Technical reference: SPF record uses "+all" qualifier — examplesupplier.com DNS TXT. Recommended: replace with "~all" or "-all".
Observed on: root domain (platform default).

As part of our due diligence process, we need to understand which third parties your platform relies on to deliver its service, particularly those that may handle the organisation data.
